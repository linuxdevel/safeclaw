# Feature 9: Directory-Scoped Sessions

## Problem

Sessions are identified by `"cli:local"` -- a hardcoded peer identity with no directory component. Running `safeclaw` in `/home/user/project-a` and `/home/user/project-b` produces the same session. Conversation history from one project bleeds into another.

## Design

### Approach

Directory-scoped sessions with project-local storage. The resolved CWD determines which session scope is active. On disk, sessions live in `<cwd>/.safeclaw/sessions/` alongside the project code. Global config (vault, safeclaw.json) stays in `~/.safeclaw/`.

### Storage Layout

```
/home/user/project-a/
├── .safeclaw/
│   ├── .gitignore               # Auto-created: "*"
│   └── sessions/
│       ├── <uuid>.json          # Full session data (history + metadata)
│       └── context-summary.json # Compressed summary of prior sessions
├── src/
└── ...
```

- `.safeclaw/` is auto-created in CWD on first run
- `.safeclaw/.gitignore` containing `*` is auto-created so the directory is always excluded from version control
- File permissions: 0o700 for directories, 0o600 for session files

### Session Lifecycle

**Startup (default -- no `--session` flag):**
1. Resolve CWD, ensure `.safeclaw/sessions/` exists
2. Load `context-summary.json` if it exists -- inject as a system-level context message (e.g., `"[Project context from previous sessions]\n<summary>"`)
3. Create a new session with a fresh UUID
4. Print `Session: <uuid>` at startup

**Startup (with `--session <id>`):**
1. Look for `<cwd>/.safeclaw/sessions/<id>.json`
2. If found, load and resume in full (complete message history)
3. If not found, error: `"Session <id> not found in <cwd>/.safeclaw/sessions/"`

**During conversation:**
- Normal agent loop; context compaction fires automatically at 80% token threshold (existing behavior)
- Parallel agents (future Feature 6) share the same `Session` object in-process

**On exit:**
1. Save the session to `<cwd>/.safeclaw/sessions/<uuid>.json`
2. Generate a context summary from the completed session using `ContextCompactor` and save to `context-summary.json` (replaces previous summary)
3. Prune: delete all session files except the most recent one (configurable, default keep=1)
4. Print `Session: <uuid>` so user can resume with `--session <uuid>`

### Context Summary

The context summary is a compressed representation of prior work, generated by the `ContextCompactor`:

```json
{
  "generatedAt": "2026-03-06T14:30:00Z",
  "sourceSessionId": "abc-123",
  "summary": "Previously worked on adding Landlock filesystem rules..."
}
```

- Generated on exit by sending the full session history to the LLM with a summarization prompt
- Loaded on next startup as a user message prepended before the conversation
- Target size: ~500-1000 tokens
- Not loaded when resuming with `--session` (the full history is already present)

### Code Changes

**`packages/core/src/sessions/types.ts`:**
- Add `scope?: string` to `PeerIdentity` (the resolved CWD path)
- Add `cwd?: string` to `SessionMetadata`

**`packages/core/src/sessions/store.ts` (FileSessionStore):**
- No interface change needed -- constructor already accepts `baseDir`; callers pass `<cwd>/.safeclaw` instead of `~/.safeclaw`

**`packages/core/src/sessions/manager.ts`:**
- Extend peer key format to `{channelId}:{peerId}:{scope}` when scope is present
- Add `pruneOldSessions(keep: number)` method

**`packages/cli/src/cli.ts`:**
- Add `--session <id>` CLI flag
- Capture `process.cwd()` and pass through to bootstrap
- On startup: print session ID
- On exit: save session, generate summary, prune old sessions, print session ID

**`packages/cli/src/commands/bootstrap.ts`:**
- Accept `cwd` parameter
- Pass `<cwd>/.safeclaw` as `FileSessionStore` base directory
- Auto-create `.safeclaw/` and `.safeclaw/.gitignore` if they don't exist

**`packages/core/src/agent/compactor.ts`:**
- Add `generateSummary(history)` public method (extracted from existing `compact()` logic) that returns just the summary text without modifying the session

**`packages/cli/src/commands/chat.ts`:**
- Load `context-summary.json` on startup if it exists and no `--session` flag
- Inject as first message in the conversation

### What Stays the Same

- **Vault**: global at `~/.safeclaw/vault.json` -- secrets are user-level
- **Config**: global at `~/.safeclaw/safeclaw.json` -- model/prompt config is user-level
- **Agent/ToolOrchestrator/Sandbox**: no changes -- stateless w.r.t. sessions
- **ProcessManager**: stays global (Feature 6 addresses per-session scoping)
- **AuditLog**: stays in-memory, global
- **Gateway/WebChat**: unchanged -- they have their own peer identities and continue using `~/.safeclaw`

### Security Considerations

- Session files contain full conversation history including tool outputs. File permissions (0o600) prevent other users from reading them.
- The `.safeclaw/` directory should not be committed to version control. The auto-created `.gitignore` prevents this by default.
- Context summaries are generated by the LLM and could theoretically leak sensitive content from prior sessions into new ones. This is by design -- the summary provides continuity. Users who want a clean start can delete `context-summary.json`.
